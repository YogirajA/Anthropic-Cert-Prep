Exam Reference: Official Scope & Structure

Source: Anthropic CCA-F Exam Guide v0.1 (Feb 10 2025) + 2026 doc refresh. Scaled score 100-1000, pass at 720. Multiple choice, 1 correct + 3 distractors. Unanswered = incorrect (no guessing penalty).
Domain weights: D1 Agentic Architecture (27%) · D2 Tool Design & MCP (18%) · D3 Claude Code Config (20%) · D4 Prompt Engineering (20%) · D5 Context & Reliability (15%).
Six official scenarios (4 picked at random per exam):
  1. Customer Support Resolution Agent: MCP tools (get_customer, lookup_order, process_refund, escalate_to_human); 80%+ first-contact resolution target.
  2. Code Generation with Claude Code: CLAUDE.md, custom slash commands, plan mode vs direct execution.
  3. Multi-Agent Research System: Coordinator + 4 subagents (web search, document analysis, synthesis, report); cited reports.
  4. Developer Productivity: Built-in tools (Read, Write, Bash, Grep, Glob) + MCP servers; codebase exploration.
  5. Claude Code for CI/CD: Automated reviews, test generation, PR feedback; minimize false positives.
  6. Structured Data Extraction: JSON schemas, validation, edge cases, downstream integration.
In-scope (study): Agentic loop & stop_reason, multi-agent orchestration, subagent context passing, tool descriptions, MCP scopes & env vars, CLAUDE.md hierarchy, .claude/rules/ glob patterns, skills (context: fork, allowed-tools, argument-hint), plan mode, -p / --output-format json / --json-schema, tool_use schemas, tool_choice, few-shot, Batch API, lost-in-the-middle, escalation triggers, structured error propagation, scratchpad files, source provenance, confidence calibration.

Domain 1: Agentic Architecture & Orchestration (27%)

Agentic loop: Send request → check stop_reason → if tool_use, execute tool, append tool_result, repeat; if end_turn, done. Other values: max_tokens, stop_sequence, refusal, pause_turn (server-tool sampling cap, resend unchanged).
SDK rename: "Claude Code SDK" is now the Claude Agent SDK. The Task tool was renamed to Agent in Claude Code v2.1.63: both names still appear (Agent in tool_use blocks, Task in init lists). Code should accept either.
AgentDefinition fields: description (required, drives delegation), prompt (subagent's system prompt), tools, disallowedTools, model (sonnet | opus | haiku | inherit), skills, memory (user | project | local), mcpServers, maxTurns, background, effort, permissionMode. (Note: isolation and color are subagent FILE-frontmatter fields, NOT SDK AgentDefinition fields.)
Subagent isolation: Subagents do NOT inherit parent's conversation history, tool results, or system prompt. The only channel from parent to subagent is the Agent tool's prompt string. Parent must include Agent in allowedTools; subagents cannot spawn subagents.
Subagent tool / MCP inheritance: Tool access is a SEPARATE dimension from context: by default subagents inherit all tools from the parent, including MCP tools. Restrict with tools (allowlist) or disallowedTools (denylist); both accept mcp__<server>__*. Use mcpServers frontmatter to scope an MCP server to just that subagent (inline definitions connect only while the subagent runs, keeping their tool descriptions out of the parent's context). Plugin subagents ignore mcpServers, hooks, and permissionMode for security.
Effort levels: low | medium | high | xhigh | max. NEWxhigh recommended for Opus 4.7 on coding/agentic tasks. Independent of extended/adaptive thinking.
ResultMessage.subtype: success, error_max_turns, error_max_budget_usd (cost cap via maxBudgetUsd), error_during_execution, error_max_structured_output_retries. result field only populated on success.
Hooks: Deterministic out-of-context handlers in settings.json. Events: PreToolUse, PostToolUse, UserPromptSubmit, SessionStart/End, Stop, SubagentStop, PreCompact, PostCompact, Notification, plus newer PostToolBatch, PostToolUseFailure, StopFailure, InstructionsLoaded, ConfigChange.
Hook handler types: NEWcommand, http, mcp_tool, prompt (model evaluates), agent (subagent evaluates).
PreToolUse decision: hookSpecificOutput.permissionDecision = "allow" | "deny" | "ask" | "defer"; can return updatedInput to modify args. Exit code 2 = blocking error (stderr fed to Claude).
Sessions: continue resumes most recent session in cwd (no ID); resume: sessionId targets specific. forkSession: true branches history into new session ID (original untouched). Stored at ~/.claude/projects/<encoded-cwd>/<id>.jsonl.
Workflow patterns (Anthropic's 5): prompt chaining, routing, parallelization (sectioning + voting), orchestrator-workers, evaluator-optimizer. Three core principles: simplicity, transparency, agent-computer interface.
Enforcement vs guidance: Use programmatic prerequisites (hooks, gates, tool_choice) when deterministic compliance is required. Prompt instructions are probabilistic; even emphatic system prompts still have a non-zero failure rate on compliance-critical operations (per PDF Task 1.4).

Domain 2: Tool Design & MCP Integration (18%)

Tool descriptions: The single biggest factor in tool selection performance. Aim for 3-4+ sentences covering what, when, when-NOT, params, edge cases. Overlapping minimal descriptions cause misrouting.
Tool definition fields: required name (regex ^[a-zA-Z0-9_-]{1,64}$), description, input_schema. Optional: NEWinput_examples, cache_control, strict: true (guarantees schema conformance), defer_loading, allowed_callers.
Structured errors: Return tool_result with is_error: true and a descriptive content string. Tool result blocks must immediately follow tool_use; results come first in user content (any text after). Claude retries 2-3x on parameter errors.
tool_choice: {"type":"auto"} (default), {"type":"any"} (force any tool), {"type":"tool","name":"..."} (force specific), {"type":"none"}. Extended/adaptive thinking is incompatible with any and named tool: only auto or none.
MCP transports: CHANGEDhttp (streamable-http, recommended for remote), sse (deprecated for remote in Claude Code), stdio (local processes). WebSocket is NOT in the 2025-11-25 MCP spec.
MCP scopes: RENAMEDlocal (default, was "project") in ~/.claude.json; project (shared via .mcp.json at repo root, version-controlled); user (was "global"). Precedence: local > project > user > plugin > connectors.
Env var substitution: ${VAR} and ${VAR:-default} in command, args, env, url, headers. Plugins additionally expose ${CLAUDE_PLUGIN_ROOT} and ${CLAUDE_PLUGIN_DATA}.
MCP tool naming: mcp__<server>__<tool> (double underscores). allowedTools accepts wildcards: mcp__github__*. Reserved name: workspace cannot be used as a server name.
Tool granularity: 4-5 tools per agent target. Consolidate related ops into one tool with an action param rather than fragmenting into create_pr/review_pr/merge_pr; but avoid kitchen-sink tools too. Scope tools to role.
Built-in API tools: text_editor_20250728 (commands: view, str_replace, create, insert; CHANGEDundo_edit removed; new max_characters param). computer_20251124 (Opus 4.5+/Sonnet 4.6, adds zoom action). Server tools: web_search_20260209, web_fetch, code_execution, tool_search.
Claude Code built-ins: Grep (ripgrep content search), Glob (filename patterns), Read (full file incl. PDF/images/notebooks), Edit (unique-text replacement), Write (create/overwrite), Bash, Agent (was Task), WebFetch, WebSearch, ToolSearch (loads schemas on demand).
Resources vs tools vs prompts (MCP): Resources = addressable read-mostly data (URIs, schemas, catalogs). Tools = actions/computations. Prompts = user-triggered templated workflows. Sampling = server asks client to call LLM (requires user approval).

Domain 3: Claude Code Configuration & Workflows (20%)

CLAUDE.md hierarchy: Managed policy (OS app-support dir, can't be excluded) → Project (./CLAUDE.md or ./.claude/CLAUDE.md, shared) → User (~/.claude/CLAUDE.md, personal) → Local (./CLAUDE.local.md, gitignored). Files concatenate, not override. Delivered as user message after system prompt.
@import: @path/to/file recursive max depth 5. Approval dialog on first encounter. claudeMdExcludes glob array skips ancestor files (managed cannot be excluded).
Auto memory: NEWv2.1.59+. Stored at ~/.claude/projects/<repo>/memory/MEMORY.md + topic files. First 200 lines / 25 KB load each session. Toggle with autoMemoryEnabled.
Skills (now the primary primitive): CHANGEDCustom slash commands have been merged into Skills. .claude/skills/<name>/SKILL.md creates /name. Old .claude/commands/*.md still works; on collision the skill wins.
SKILL.md frontmatter: name, description (model uses to decide), when_to_use, argument-hint, arguments, disable-model-invocation, user-invocable, allowed-tools, model, effort, context: fork, agent, hooks, paths. Substitutions: $ARGUMENTS, $N, ${CLAUDE_SKILL_DIR}, ${CLAUDE_SESSION_ID}.
Skill locations / precedence: Enterprise > Personal (~/.claude/skills/) > Project (.claude/skills/). Plugin skills always namespaced as plugin-name:skill-name. context: fork isolates verbose output in a subagent.
Sub-agents: .claude/agents/<name>.md (project) or ~/.claude/agents/ (user) or plugin. Frontmatter: name, description, tools, disallowedTools, model, permissionMode, mcpServers, hooks, maxTurns, skills, memory, effort, background, isolation, color. Built-in: Explore, Plan, general-purpose.
Plugins: NEWManifest at <plugin>/.claude-plugin/plugin.json. Components at plugin root: skills/, commands/, agents/, hooks/hooks.json, .mcp.json, bin/. Distribute via marketplace, git URL, --plugin-dir, --plugin-url. Use ${CLAUDE_PLUGIN_ROOT}.
.claude/rules/: Modular rule files with optional YAML paths: glob frontmatter (e.g., **/*.test.tsx). Loads only when matching files touched. Better than directory-level CLAUDE.md for cross-directory patterns.
Plan mode: Use for large-scale changes, multi-file mods, architectural decisions. Activate via --permission-mode plan or Shift+Tab. ExitPlanMode tool returns plan; user must accept.
CI / Headless: -p / --print for non-interactive. NEW--bare skips hooks/skills/plugins/MCP/auto-memory/CLAUDE.md (recommended for CI; will become default for -p). --output-format json | stream-json. --json-schema '<schema>' → result in structured_output field.
Permission modes: default, acceptEdits (auto-approves edits + filesystem commands such as mkdir/touch/mv/cp; other shell commands need explicit allow rules), plan, auto (TS classifier), NEWdontAsk (denies anything not in permissions.allow: strictest CI mode), bypassPermissions.

Domain 4: Prompt Engineering & Structured Output (20%)

Be specific, not vague: "Flag when claimed behavior contradicts code" beats "be conservative." Claude 4.x is more literal: vague hedges like "be thorough" or "high-confidence only" cause recall drops because the model now obeys.
Multishot examples: UPDATED3-5 diverse examples (was 2-4), wrapped in <example> tags inside <examples>. Show ambiguous-case reasoning. Most effective when judgment matters most.
XML tags: Anthropic-canonical structure: <instructions>, <context>, <input>, <documents>, <document index="n">, <document_content>, <source>, <thinking>, <answer>, <quotes>.
Long context: Place documents at the TOP, query at the END (~30% quality lift on multi-doc). Ground the model in <quotes> first.
Adaptive thinking (replaces budget_tokens): NEWthinking: {type: "adaptive"} + output_config.effort: low|medium|high|xhigh|max. Manual extended thinking (budget_tokens) is NOT supported on Opus 4.7, Opus 4.8, or Fable 5 / Mythos 5 (returns 400). Interleaved thinking is automatic on Opus 4.7/4.8/4.6. On Fable 5 / Mythos 5 thinking is always on (omit the param; even {type:"disabled"} 400s), whereas on Opus 4.7/4.8 adaptive is opt-in (off unless you set {type:"adaptive"}).
Prefilling deprecated: CHANGEDLast-assistant-turn prefill is no longer supported on Claude 4.6 / 4.7 / 4.8 / Fable 5 / Mythos 5 (returns 400). Migrate to Structured Outputs, explicit "respond directly" instructions, XML wrappers, or tool calling.
Structured Outputs (GA 2026): NEWNative JSON-schema enforcement via output_config.format. SDK helpers: client.messages.parse() with Pydantic; zodOutputFormat(); outputConfig(Class<T>). Limits: 20 strict tools max, 24 optional params, 16 union-typed params. Incompatible with Citations (400 error).
Strict tool use: strict: true on a tool guarantees schema-valid input. Combine with tool_choice: "any" to guarantee both a call and a valid schema. Make optional fields nullable to prevent hallucination.
Schema redundancy: Include calculated_total + stated_total; flag mismatch for human review. Catches OCR/extraction errors automatically.
Resilient enums: Add "other" + detail string field. Avoid fragile expansion as new types arrive. For amended docs, capture multiple values with source and effective_date.
Batch API: 50% discount, 100,000 requests OR 256 MB per batch (whichever first), 24-hour expiry, results retained 29 days. custom_id for correlation. NOT ZDR-eligible. Supports vision, tool use, multi-turn, thinking, structured outputs. Use 1-hour cache (5-min often expires mid-batch).
Citations: citations: {enabled: true} on each document block (all-or-none). Returns char_location, page_location, or content_block_location. cited_text is free. Works with prompt caching and Batch API. ZDR-eligible.

Domain 5: Context Management & Reliability (15%)

1M context (GA, no beta): NEWOpus 4.8, Opus 4.7, Opus 4.6, Sonnet 4.6, and Fable 5 / Mythos 5 ship with a 1M context window at standard pricing (no long-context premium). Haiku 4.5 = 200k. Sonnet 4.5 1M beta retired in 2026.
Lost-in-the-middle: Models reliably process start/end of long inputs but may miss findings in middle sections. Place key summaries first; use explicit section headers; ground in quotes.
Prompt caching: Up to 4 explicit breakpoints per request. Hierarchy: tools → system → messages (changes invalidate that level and all subsequent). Place cache_control on the LAST stable block.
Cache TTL: 5-min default ({"type":"ephemeral"}) or 1-hour ({"type":"ephemeral","ttl":"1h"}). 1h breakpoints must appear before 5m in the prompt. Multipliers: 5m write 1.25x, 1h write 2x, reads 0.1x.
Cache minimums: Opus 4.7 / 4.6 / 4.5 = 4096 tokens; Haiku 4.5 = 4096 tokens; Sonnet 4.6 = 2048 tokens; Sonnet 4.5 = 1024 tokens. Below-minimum caches silently with no error: verify via cache_creation_input_tokens / cache_read_input_tokens in usage. (Caveat: prompt-caching minutiae is OUT of exam scope, and exact per-model floors have shifted — a Jun-2026 docs check showed newer models with lower floors, e.g. Opus 4.8 = 1024, Opus 4.7 = 2048. Don't memorize; just know below-min caching silently no-ops.)
Memory tool (GA Feb 17, 2026): NEW{"type": "memory_20250818", "name": "memory"}: no beta header. Client-side: YOU implement file backend restricted to /memories. Commands: view, create, str_replace, insert, delete, rename. Validate paths to block ../ traversal.
Server-side compaction (beta): NEWHeader compact-2026-01-12. Supported on Opus 4.8/4.7/4.6, Sonnet 4.6, and Fable 5 / Mythos 5. API drops everything before the compaction block on next turn. Pairs with memory tool (compaction summarizes; memory persists facts).
Context editing (separate beta): Header context-management-2025-06-27. Strategies: clear_tool_uses_20250919, clear_thinking_20251015. All Claude models supported.
Context preservation patterns: Extract transactional facts into a persistent "case facts" block. Trim verbose tool outputs (40+ fields → ~5 relevant) before they hit context. Use scratchpad files for extended sessions to counteract context degradation.
Escalation triggers: Customer explicitly requests human, policy is silent/ambiguous, agent unable to make progress. NOT sentiment analysis, NOT self-reported confidence (poorly calibrated).
Error propagation: Return structured failure context (failure type, attempted query, partial results, alternative approaches). Distinguish access failures (timeout, perm denied) from valid empty results. Never silently suppress errors.
pause_turn handling: Server tools (web_search, web_fetch, code_execution) hit a 10-iteration server-side loop cap and return pause_turn. Re-send the assistant response unchanged to continue.
Confidence calibration: Validate accuracy by document type AND field before reducing review. Aggregate (97%) can mask poor performance on specific segments. Stratified random sampling.
Provenance: Subagents output structured claim-source mappings (claim, evidence, source, confidence, dates). Synthesis preserves attributions. Annotate temporal conflicts (2023 vs 2024 stat) instead of arbitrarily picking one or averaging.

Architect's Playbook: Design Patterns & Anti-Patterns (mix of PDF + third-party)

4 Constraint Dimensions (third-party framing): Latency → parallelization & caching. Accuracy → structured intermediates & few-shot. Cost → Batch APIs & context pruning. Compliance → application-layer intercepts (NEVER prompts). Anthropic explicitly names only the latency/cost tradeoff in "Building Effective Agents"; accuracy and compliance are scaffolding added by the third-party Playbook.
Zero-Tolerance Compliance: Emphatic prompts have a non-zero failure rate. Use hooks (PreToolUse → permissionDecision: "deny") or tool_choice to enforce. Block server-side, invoke escalation. Remove model discretion.
Tool Context Pruning: Filter verbose tool responses (40+ fields → ~5 relevant) application-side before they enter context. Essential for multi-issue support sessions.
Session Resumption: Filter out stale tool_result messages on resume; keep human/assistant turns. Forces agent to re-fetch current data. For dynamic environments, inform agent which files changed.
Scratchpad Pattern: Persist key findings, architectural maps, decisions in scratchpad.md. Reference for subsequent questions. Counteracts context degradation in 30+ minute sessions. Pairs with the Memory tool for cross-session persistence.
Lead-Coordinator State Management (corrected from Anthropic's multi-agent research blog): Subagents do NOT share memory or read each other's state. They return findings to the lead/coordinator only. The LEAD persists its plan to memory/filesystem to survive long contexts; subagents write outputs to a filesystem and pass lightweight references back to the lead to minimize information loss during multi-stage processing (the blog's framing). Avoid daisy-chaining full conversation logs (which scales tokens exponentially): pass concise structured findings instead. Note: the third-party "shared vector store between subagents with semantic search" is NOT Anthropic-recommended; their architecture is lead-coordinator-only.
Forced Execution Order: Use tool_choice: {"type":"tool","name":"extract_metadata"} on first API call. Don't rely on prompt begging: use API constraints. Note: incompatible with adaptive/extended thinking.
Goal-Oriented Delegation: Specify goals + quality criteria, not step-by-step procedures. Let subagents determine strategy. Add enum parameters (analysis_type: extraction | summarization) to guide behavior.
MCP Tool Granularity (right-size, don't over-split): Anthropic's "Writing Tools for Agents" recommends consolidating related ops (e.g., schedule_event combining list_users/list_events/create_event). Split into focused tools when descriptions overlap; consolidate when steps belong to one workflow. Improving descriptions often beats splitting. Agents default to familiar built-ins (Grep) when custom tools are vaguely described.
Structured Intermediates: Standardize multi-format subagent outputs to common JSON (claim, evidence, source, confidence) via format conversion layer. Preserve claim-source mappings through synthesis.
Schema Redundancy: calculated_total + stated_total. Flag mismatch for human review. Catches OCR and extraction errors automatically.
Resilient Catch-Alls: Enum + "other" + detail string. Avoid fragile enum expansion. For amended docs, capture multiple values with source and effective_date.
Memory + Compaction Pattern: 2026For long-running agents: enable server-side compaction beta (summarizes prior turns) + memory tool (writes durable facts to /memories/). Compaction handles "what happened"; memory handles "what to remember next time."
1M Context Caveats: 20261M GA on Opus 4.7/4.6 and Sonnet 4.6 with no premium. Still subject to lost-in-the-middle and context rot. Use prompt caching + just-in-time retrieval. Don't dump everything; load on demand.
8 Agent Design Patterns (framework view): Common framework names mapped to Anthropic implementation. Useful as study scaffolding; the CCA-F exam uses Anthropic's own terminology, not these names (only "scratchpad" and "plan mode" appear in the official guide by name).
  Layer 1: Reasoning primitives (single LLM call):
    1. Scratchpad → persist findings to scratchpad.md or memory tool's /memories/. Anthropic name: scratchpad files / memory tool.
    2. Chain of Thought → <thinking>/<answer> XML tags or extended/adaptive thinking. Anthropic name: extended thinking / adaptive thinking.
    3. Self-Consistency → sample N completions, take majority. Not in CCA-F by name; closest: independent reviewer + multi-pass review.
  Layer 2: Agent loop patterns (loop, tools, persistent state):
    4. ReAct → reason then call tool, repeat. Anthropic name: agentic loop driven by stop_reason: tool_use.
    5. Plan & Execute → produce a plan up front, then execute. Anthropic name: plan mode (Shift+Tab; --permission-mode plan; ExitPlanMode tool).
    6. Reflexion → CoT applied to the agent's own past trace; critique and re-plan. Anthropic name: independent reviewer / multi-pass review / evaluator-optimizer workflow.
    7. Interleaved Thinking → thinking block between every tool call. Anthropic name: interleaved thinking (automatic on Opus 4.7/4.6; Sonnet 4.6 needs interleaved-thinking-2025-05-14 beta header).
    8. Agent Memory → durable state across sessions. Anthropic name: memory tool (memory_20250818, GA) + auto memory (Claude Code v2.1.59+) + agent memory: user|project|local frontmatter.
Key insight: the loop layer always contains the primitive layer inside it. ReAct = Chain of Thought + tool call. Reflexion = Chain of Thought applied to the agent's own trace.

Permission Modes

Permission modes set a baseline for what runs without a prompt. Cycle with Shift+Tab in the CLI (default → acceptEdits → plan; optional modes slot in after), or set with --permission-mode <mode>.

Context Window & "Lost in the Middle"

Claude Code's context window holds the system prompt, conversation history, tool results, CLAUDE.md, and active skills. How it's filled matters as much as how big it is.

What "lost in the middle" means

LLMs exhibit a U-shaped attention curve. Information at the start and end of the context is recalled reliably; information buried in the middle of a long context is often missed, even though it is technically present.

Practical impact: A critical instruction placed at message 40 of an 80-message conversation may simply be ignored. The token is in context, but attention is not on it.

How Claude Code mitigates it

MechanismWhat it does
System prompt at topCore rules sit at the start where attention is strongest.
System remindersRe-injected near the end of context — surface skills, deferred tools, env right before the model responds.
Auto-compactionAs context fills, older turns are summarized so the working window stays dense with relevant info.
Subagent isolationDelegating research to a subagent keeps the parent context lean; only the result returns.
CLAUDE.mdProject memory loaded at the top of every session — guaranteed high-attention placement.
Tool result stripping (auto mode)The classifier doesn't see raw tool results, only your messages and tool calls — limits prompt-injection blast radius.

What you can do

  • Restate critical constraints near the end of long sessions — don't assume page-1 instructions still rule.
  • Use subagents for big searches so raw results don't bloat your main context.
  • Compact deliberately with /compact before starting a new sub-task.
  • Prefer plans & todos over long chat history — they're re-read each turn and stay salient.
  • Use /clear between unrelated tasks rather than letting context drift.
  • For hard guarantees, use deny rules — stated boundaries in chat ("don't push") can be lost if compaction removes the message that stated them.

Token budgeting rule of thumb

Window usageBehavior
< 50%Healthy — full recall
50–80%Middle-of-context risk rising; consider compacting
> 80%Auto-compaction imminent; expect loss of nuance from older turns

Best practices (top of mind)

1Pick the right transport & scope: http (streamable-http) for remote, stdio for local. Shared servers in .mcp.json at repo root.Precedence is local > project > user > plugin > connectors. Wrong scope leaks personal config into team or vice versa.

4. Transient error with retry hint

{
  "content": [
    { "type": "text", "text": "Upstream rate limit hit (429). Retry after 4s." }
  ],
  "isError": true,
  "_meta": {
    "errorCode": "RATE_LIMITED",
    "retryable": true,
    "retryAfterSeconds": 4,
    "attemptBudget": 3
  }
}
Why this shape: isError: true with a short, actionable message; the _meta block carries machine-readable retry guidance. Claude defaults to 2-3 retries on parameter errors, but explicit retryAfterSeconds lets it back off on 429/503 instead of hammering and burning budget.

5. Permanent error (no retry)

{
  "content": [
    { "type": "text", "text": "order_id 'A-99' is malformed: expected pattern ^A-[0-9]{4,8}$. Fix the argument and call again." }
  ],
  "isError": true,
  "_meta": {
    "errorCode": "INVALID_ARGUMENT",
    "retryable": false,
    "field": "order_id"
  }
}
Why this shape: permanent failures (validation, auth, not-found) say retryable: false and tell Claude how to fix the call. Without that, the agent wastes turns re-trying the same broken arguments.

Quick answers to common confusions

"Can I set max_tokens: 0 in Claude Code to warm the cache?" No — that's an API-only pattern. In Claude Code the cache is warmed automatically by the system prompt + CLAUDE.md + tool definitions, which stay stable across turns.
"Can I switch cache TTL to 1 hour in Claude Code?" No — Claude Code uses the default 5-minute sliding cache. The 1-hour TTL beta is API-only.
"How do I set temperature in Claude Code?" You can't. Sampling parameters are fixed by the harness. If you need them, use the API directly or the Agent SDK.
"Can I see cache hit rates per turn in Claude Code?" Not directly. Use /cost for session totals, or hit the API directly to inspect usage.cache_read_input_tokens.
"Hooks vs Skills vs Plugins — which is in the API?" None of them. All three are Claude Code constructs. The API is stateless — hooks/skills/plugins live in the client harness.
Bridging the two: the Claude Agent SDK exposes a programmatic Claude Code (skills, hooks, MCP, subagents) on top of the API — use it when you want Claude-Code-like behavior inside your own application instead of the CLI.

Known Distractors (Common Wrong Answers)

Patterns that look right but are almost always the trap. Synthesized from passing-candidate write-ups, the agilefever CCA-F anti-pattern guide, Sachin Trivedi's exam recap, Rick Hightower's 60-question practice set, the Big Tech Careers distractor taxonomy, and the Anthropic course curriculum. If an answer choice matches one of these, it's almost certainly wrong.

The meta-rule that eliminates ~70% of wrong answers: "If it must always happen, use code. If it should usually happen, use a prompt." Prompts ask Claude to do something; hooks / tool_choice / programmatic gates enforce it. The exam consistently rewards deterministic over probabilistic, root cause over symptom, and the simplest architectural fix over the most sophisticated one.

Quick decision rules to defuse distractors

  1. "Must always happen" answers → pick code / hook / gate, not prompt.
  2. "Larger model" or "more compute" → almost never the right answer. Look for the architectural fix instead.
  3. "Parse text" / "check content" → wrong. Use stop_reason.
  4. "Add few-shot" when the question is about tool behavior → wrong. Improve tool descriptions first.
  5. "Sentiment / confidence score" as a trigger → wrong. Explicit policy triggers.
  6. "One mega-tool" or "10+ tools" → both wrong. ~4–5 with action consolidation.
  7. "One mega-agent" → wrong. Decompose with hub-and-spoke.
  8. "Same session reviews its own work" → wrong. Independent instance.
  9. "Run N times, pick best" → wrong. Structured passes.
  10. "Inject everything into the system prompt" → wrong. Retrieval / scoping.
  11. "Increase temperature for creativity / lower for accuracy" → usually wrong. Fix the design.
  12. "Silently swallow errors" → wrong. Structured failure context.
  13. "Batch for real-time" or "sync for nightly bulk" → mismatched API choice.
  14. ".gitignore = security" → wrong. permissions.deny.

Keep a session running — 3 approaches

The docs frame /loop, /goal, and Stop hooks as three ways to make Claude keep working past a single turn. They differ by what drives the continuation:

ApproachDriven byUse whenSyntax
/loopTime — a recurring interval"Do X every N minutes" / poll something on a cadence/loop 30m <prompt>
/goalA condition being met"Keep going until X is true," done is unknown # of turns away/goal <condition>
Stop hookCustom code you writeYou need programmatic / deterministic control of when to stopStop hook in settings.json

Mental model: /loop is a metronome, /goal is a finish line, a Stop hook is a referee. (/goal is in fact implemented as a managed prompt-based Stop hook.)

/loop — recurring runs on an interval

A bundled skill that re-runs a prompt automatically while the session stays open. Both the interval and the prompt are optional, and what you pass changes the behavior:

  • /loop 5m check the deploy and ping me on failure — fixed interval (interval + prompt).
  • /loop check the deploy — self-paced (dynamic) mode: Claude picks a delay between 1 min and 1 hour each iteration based on observed activity.
  • /loop — runs the built-in maintenance prompt (or your .claude/loop.md) on a dynamically chosen interval.
  • /loop 20m /review-pr 1234 — can re-run another slash command / skill each iteration.

Interval units: s / m / h / d (lead token 30m or trailing clause every 2 hours). It compiles the interval to a cron job, prints a job ID, and auto-expires after 7 days. Stop it with Esc. Default behavior is customizable via .claude/loop.md or ~/.claude/loop.md. Requires Claude Code v2.1.72+; disable with CLAUDE_CODE_DISABLE_CRON=1.

/goal — work until a condition is met

Sets a completion condition; Claude keeps working across turns until a small fast model (Haiku by default) confirms the condition holds in the transcript, then the goal auto-clears. Unlike /loop, it is condition-driven, not time-driven.

  • /goal <condition> — set a goal and start a turn immediately (condition up to 4,000 chars).
  • /goal — show status (turns used, tokens, the evaluator's last reason).
  • /goal clear — cancel early (aliases: stop, off, reset, none, cancel).
  • claude -p "/goal CHANGELOG has an entry for every PR merged this week" — works headless too.

It is a wrapper around a session-scoped prompt-based Stop hook: the evaluator only judges what is already in the transcript (it does not run tools or read files itself), so phrase conditions in terms of what the session will have produced. One goal per session; restored on --resume/--continue. Requires v2.1.139+ and an accepted trust dialog (unavailable when disableAllHooks / allowManagedHooksOnly is set).
