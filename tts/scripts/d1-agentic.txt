Domain 1, Agentic Architecture and Orchestration.

Section one, cheat sheet.

Domain 1: Agentic Architecture & Orchestration. Agentic loop: Send request then check stop_reason then if tool_use, execute tool, append tool_result, repeat; if end_turn, done. Other values: max_tokens, stop_sequence, refusal, pause_turn (server-tool sampling cap, resend unchanged). SDK rename: "Claude Code SDK" is now the Claude Agent SDK. The Task tool was renamed to Agent in Claude Code v2.1.63: both names still appear (Agent in tool_use blocks, Task in init lists). Code should accept either. AgentDefinition fields: description (required, drives delegation), prompt (subagent's system prompt), tools, disallowedTools, model (sonnet or opus or haiku or inherit), skills, memory (user or project or local), mcpServers, maxTurns, background, effort, permissionMode. (Note: isolation and color are subagent FILE-frontmatter fields, NOT SDK AgentDefinition fields.). Subagent isolation: Subagents do NOT inherit parent's conversation history, tool results, or system prompt. The only channel from parent to subagent is the Agent tool's prompt string. Parent must include Agent in allowedTools; subagents cannot spawn subagents. Subagent tool / MCP inheritance: Tool access is a SEPARATE dimension from context: by default subagents inherit all tools from the parent, including MCP tools. Restrict with tools (allowlist) or disallowedTools (denylist); both accept mcp__<server>__*. Use mcpServers frontmatter to scope an MCP server to just that subagent (inline definitions connect only while the subagent runs, keeping their tool descriptions out of the parent's context). Plugin subagents ignore mcpServers, hooks, and permissionMode for security. Effort levels: low or medium or high or xhigh or max. xhigh recommended for Opus 4.7/4.8 and Fable 5 / Mythos 5 on coding/agentic tasks. Independent of extended/adaptive thinking. ResultMessage.subtype: success, error_max_turns, error_max_budget_usd (cost cap via maxBudgetUsd), error_during_execution, error_max_structured_output_retries. result field only populated on success. Hooks: Deterministic out-of-context handlers in settings.json. Events: PreToolUse, PostToolUse, UserPromptSubmit, SessionStart/End, Stop, SubagentStop, PreCompact, PostCompact, Notification, plus newer PostToolBatch, PostToolUseFailure, StopFailure, InstructionsLoaded, ConfigChange. Hook handler types: command, http, mcp_tool, prompt (model evaluates), agent (subagent evaluates). PreToolUse decision: hookSpecificOutput.permissionDecision = "allow" or "deny" or "ask" or "defer"; can return updatedInput to modify args. Exit code 2 = blocking error (stderr fed to Claude). Sessions: continue resumes most recent session in cwd (no ID); resume: sessionId targets specific. forkSession: true branches history into new session ID (original untouched). Stored at about /.claude/projects/<encoded-cwd>/<id>.jsonl. Workflow patterns (Anthropic's 5): prompt chaining, routing, parallelization (sectioning + voting), orchestrator-workers, evaluator-optimizer. Three core principles: simplicity, transparency, agent-computer interface. Enforcement vs guidance: Use programmatic prerequisites (hooks, gates, tool_choice) when deterministic compliance is required. Prompt instructions are probabilistic; even emphatic system prompts still have a non-zero failure rate on compliance-critical operations (per PDF Task 1.4).

Section two, quick review.

Stop reasons. tool_use then loop, end_turn then done, pause_turn then resend assistant unchanged, max_tokens (last block tool_use) then raise budget, refusal then reword, stop_sequence then inspect.

Loop control. Check stop_reason programmatically. Tool results are appended to history so the model reasons about the next action.

Subagent context. NO inheritance of conversation, tool results, or parent system prompt. The ONLY channel from parent is the Agent tool's prompt string.

Subagent tool/MCP inheritance. Separate dimension from context: subagents DO inherit all parent tools by default, including MCP tools. Restrict via tools (allowlist) or disallowedTools (denylist) - both accept mcp__<server>__*. Use mcpServers frontmatter to scope a server to just that subagent (inline defs connect only during the run, keeping tool descriptions out of parent context). Plugin subagents IGNORE mcpServers, hooks, permissionMode.

Spawning subagents. Use the Agent tool (renamed from Task in v2.1.63: both names valid). Parent's allowedTools must include it.

Subagent rules. Cannot spawn other subagents. Don't include Agent in their tools. Each subagent has its own AgentDefinition.

AgentDefinition. description (drives delegation), prompt, tools, disallowedTools, model, skills, memory, mcpServers, initialPrompt, maxTurns, background, effort, permissionMode. Note: isolation and color are sub-agent FILE frontmatter (.claude/agents/), not AgentDefinition fields.

Effort levels. low or medium or high or xhigh or max. xhigh recommended for coding/agentic on Opus 4.7/4.8 and Fable 5. Haiku 4.5 has no xhigh/max.

Managed Agents (context only). Beta server product (header managed-agents-2026-04-01): Anthropic runs the loop + a sandbox environment per session (cloud OR self-hosted). Create versioned POST /v1/agents, then POST /v1/sessions; output streams over SSE. Contrast: Agent SDK runs the loop in YOUR infra. Not on the blueprint — know the SDK-vs-Managed distinction, not the API surface.

isolation: worktree. Subagent file frontmatter only (.claude/agents/*.md), NOT an SDK AgentDefinition field. Spins up a temporary git worktree per run so parallel agents don't clobber each other's edits; auto-removed if the subagent makes no changes. Branches from origin/HEAD unless worktree.baseRef:"head".

Two isolations — don't confuse. Context isolation (always): a subagent gets its own context window; only the Agent-tool prompt string flows in, only the result flows back. Worktree isolation (opt-in via isolation: worktree): a separate git working dir so file edits don't collide. Independent mechanisms.

ResultMessage subtype. success or error_max_turns or error_max_budget_usd or error_during_execution or error_max_structured_output_retries. result only populated on success.

Hooks vs prompts. Hooks are DETERMINISTIC (out-of-context handlers). Prompts probabilistic: emphatic system prompts have a non-zero failure rate on compliance.

Hook events. PreToolUse, PostToolUse, UserPromptSubmit, SessionStart/End, Stop, SubagentStop, PreCompact, Notification, PostToolBatch.

Hook handler types. command, http, mcp_tool, prompt, agent (model-evaluated).

PreToolUse decisions. allow or deny or ask or defer. Exit code 2 = block + stderr to Claude. updatedInput can modify args before tool runs.

PostToolUse. Intercepts results before model sees. decision: 'block' halts loop with reason. CANNOT undo the tool action.

Sessions. continue then most recent in cwd (no ID). resume: id then specific session. forkSession: true then branch history into new ID (original untouched).

Session storage. about /.claude/projects/<encoded-cwd>/<id>.jsonl.

Workflow patterns. Prompt chaining, routing, parallelization (sectioning + voting), orchestrator-workers, evaluator-optimizer.

ReAct pattern. Reasoning + Acting loop: model thinks, calls a tool, observes result, repeats. In Claude this is the canonical agentic loop driven by stop_reason = tool_use. Layer-2 design pattern; the reasoning step uses <thinking> tags or adaptive thinking.

Plan and Execute. Generate full plan first, then execute steps with minimal further reasoning. In Claude Code: Plan mode (Shift+Tab or --permission-mode plan); ExitPlanMode tool returns the plan; user approves before execution.

Reflexion / self-critique. Apply chain-of-thought to the agent's OWN trace: review past actions, critique, re-plan. In Claude: independent reviewer instance, multi-pass review, evaluator-optimizer workflow. Anthropic's recommendation: a separate session catches more than self-review.

Anthropic principles. Simplicity, transparency, agent-computer interface (ACI). Start simple.

Multi-concern requests. Decompose into items then investigate in parallel with shared context then synthesize unified response.

Coordinator role. Decompose + delegate + aggregate + route errors. Specify GOALS and quality criteria, not step-by-step procedures.

Parallel subagents. Emit multiple Agent tool calls in a single coordinator response (NOT across separate turns).

Text parsing for termination. Always check stop_reason. Parsing assistant text for completion is unreliable.

Narrow decomposition. "AI in creative industries" decomposed into only visual arts misses music/writing/film. Coordinator must cover scope.

Assuming inheritance. Subagents do NOT inherit parent context. Pass it explicitly in the prompt.

Emphatic prompt for compliance. "CRITICAL: NEVER process >$500" has a non-zero failure rate. Use hooks or tool_choice to enforce.

Section three, question and answer review.

Scenario, Customer Support Resolution Agent. Question, A customer support agent receives a response with stop_reason "tool_use". What is the correct next step in the agentic loop? Answer, Execute the requested tool and send the tool_result back to the model in a new user turn. Why, stop_reason "tool_use" signals that Claude wants to call a tool. You execute it client-side and return a tool_result block in the next user message. Parsing text or ending the turn breaks the loop; resending the prompt loses tool intent.

Scenario, Multi-Agent Research System. Question, A coordinator agent invokes three subagents via the Agent tool to research separate topics. What context does each subagent automatically inherit from the coordinator? Answer, Only its AgentDefinition.prompt plus the prompt string passed via the Agent tool call. Why, Subagents start with a fresh context window. The only channel from parent to subagent is the Agent tool's prompt string; the subagent receives its own AgentDefinition.prompt plus that prompt. It does not get the parent's history, tool_results, or system prompt.

Scenario, Code Generation with Claude Code. Question, Which stop_reason indicates that the server-side sampling loop reached its iteration limit while executing server tools (e.g., web search)? Answer, pause_turn. Why, pause_turn is returned when the server-side sampling loop hits its iteration limit (default 10) during server tools like web_search. The fix is to send the assistant response back as-is to continue. max_tokens is for output truncation; tool_use is for client tool calls; stop_sequence is for custom stop strings.

Scenario, Developer Productivity Tools. Question, An architect wants to enforce that no file matching *.env can ever be read by the agent, regardless of the prompt. What is the most reliable mechanism? Answer, Configure a PreToolUse hook that returns permissionDecision: "deny" for matching tool inputs. Why, PreToolUse hooks are deterministic and run before every tool call: they can return permissionDecision "deny" to block the call. System prompts are probabilistic guidance that the model can violate. Post-hoc checks miss the read; effort levels do not enforce policy.

Scenario, Multi-Agent Research System. Question, The Anthropic guidance distinguishes "workflows" from "agents". Which definition matches a workflow? Answer, A system where LLMs and tools are orchestrated through predefined code paths. Why, Anthropic's "Building effective agents" defines workflows as systems where LLMs and tools are orchestrated through predefined code paths. Agents, by contrast, dynamically direct their own processes and tool usage.

Scenario, Multi-Agent Research System. Question, In an orchestrator-workers pattern, what makes it different from simple parallelization? Answer, The central LLM dynamically decomposes tasks; subtasks are determined by the input, not pre-defined. Why, Orchestrator-workers differs from parallelization because subtasks are not pre-defined. A central LLM dynamically decomposes the task and delegates to worker LLMs based on the input. Parallelization (sectioning) uses pre-defined splits.

Scenario, Code Generation with Claude Code. Question, A developer wants the SDK to automatically resume the most recent session in the current working directory without tracking session IDs. Which option achieves this? Answer, continue_conversation=True (Python) or continue: true (TypeScript). Why, continue_conversation=True in Python (or continue: true in TypeScript) tells the SDK to find and resume the most recent session in the current directory without an ID. resume requires a specific session_id; fork_session creates a new branch; sessionId="auto" is not a valid value.

Scenario, Customer Support Resolution Agent. Question, Which AgentDefinition field tells Claude when to delegate a task to that subagent? Answer, description. Why, The description field is the natural-language signal Claude uses to decide whether to invoke a subagent. prompt defines the subagent's behavior, model overrides the model, tools restricts capabilities. Only description drives the routing decision.

Scenario, Multi-Agent Research System. Question, Which statement about subagent recursion is correct? Answer, Subagents cannot spawn their own subagents; do not include Agent in a subagent's tools array. Why, Anthropic's docs explicitly state subagents cannot spawn their own subagents, and you should not include Agent in a subagent's tools array. This prevents infinite nesting and matches the Plan/Explore built-in pattern.

Scenario, Claude Code for Continuous Integration. Question, Which effort levels are valid for an AgentDefinition? Answer, "low", "medium", "high", "xhigh", "max". Why, The documented effort levels are low, medium, high, xhigh, and max (also accepting a number). The other options use non-standard terminology not in the SDK reference.

Scenario, Code Generation with Claude Code. Question, Which exit code from a command-type hook produces a blocking error that prevents the action and uses stderr as the reason? Answer, 2. Why, Exit code 2 is the blocking error: the action is blocked and stderr is used as the reason. Exit 0 means success (parse stdout). Exit 1 and other non-zero codes are non-blocking errors that show stderr but continue.

Scenario, Customer Support Resolution Agent. Question, A PreToolUse hook returns permissionDecision "ask". What happens? Answer, The user is prompted for confirmation. Why, "ask" prompts the user to confirm the tool call. "allow" skips the prompt, "deny" blocks the call, "defer" pauses for external handling (print mode only).

Scenario, Structured Data Extraction. Question, Where are session transcripts stored on the local filesystem by default? Answer, about /.claude/projects//.jsonl. Why, Sessions are stored under about /.claude/projects//.jsonl, where the encoded cwd replaces non-alphanumeric characters with hyphens. Resume requires a matching cwd.

Scenario, Multi-Agent Research System. Question, Which workflow pattern best fits running identical tasks multiple times for diverse outputs (e.g., multi-perspective code reviews)? Answer, Parallelization (voting). Why, The voting variant of parallelization runs identical tasks multiple times for diverse outputs. Sectioning splits into independent parallel subtasks; chaining is sequential; evaluator-optimizer is iterative critique.

Scenario, Customer Support Resolution Agent. Question, A model returns stop_reason "refusal". What is the recommended response? Answer, Recognize Claude declined for safety reasons; consider rephrasing or using a different model. Why, refusal indicates Claude declined for safety reasons. The docs recommend rephrasing, modifying, or trying a different model (e.g., Haiku). Retrying or bumping tokens does not address the cause.

Scenario, Code Generation with Claude Code. Question, What is the strongest reason to prefer hooks over prompt instructions for compliance enforcement? Answer, Hooks are deterministic; prompt instructions are probabilistic and can be ignored by the model. Why, Hooks are deterministic code paths invoked by the harness; the model cannot ignore them. Prompt instructions are probabilistic guidance the model may not follow. This is the core compliance argument for hooks.

Scenario, Structured Data Extraction. Question, In a TypeScript SDK call, you want to fork an existing session to explore an alternative approach without losing the original. Which option do you set? The fork starts with a copy of the original's history and gets a new session_id while the original is unchanged.

Scenario, Developer Productivity Tools. Question, What is the anti-pattern of relying on max_turns or maxTurns as the primary mechanism to terminate an agentic loop? Answer, It treats the iteration cap as a normal termination signal instead of an emergency safeguard; the loop should end on stop_reason "end_turn". Why, Iteration caps are emergency safeguards. The correct termination is when stop_reason becomes end_turn (or the equivalent ResultMessage). Relying on max_turns to end work means the agent never naturally completes.

Scenario, Multi-Agent Research System. Question, In the hub-spoke (coordinator-subagent) pattern, what role does the coordinator play? Answer, A central agent that decomposes tasks, delegates to subagents via the Agent tool, and synthesizes results. Why, The coordinator is the hub that decomposes tasks, dispatches them to specialized subagents (spokes) via the Agent tool, and synthesizes their final outputs. Subagents return only their final message; they cannot talk to each other directly.
