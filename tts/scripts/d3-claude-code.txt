Section two, quick review.

CLAUDE.md hierarchy. Managed (OS app-support, can't exclude) then Project (./CLAUDE.md or ./.claude/CLAUDE.md) then User (about /.claude/CLAUDE.md) then Local (./CLAUDE.local.md). FILES CONCATENATE.

Project vs user. Project = shared via VCS. User-level (about /.claude/) is PERSONAL ONLY: not shared with teammates.

@import. @path/file recursive max 5. Approval dialog on first encounter. claudeMdExcludes can skip ancestor files (managed cannot be excluded).

Auto memory. about /.claude/projects/<repo>/memory/MEMORY.md + topic files. First 200 lines / 25 KB load each session. v2.1.59+.

Skills (now primary). .claude/skills/<name>/SKILL.md creates /name. Custom commands merged into Skills. Old .claude/commands/*.md still works.

Skill frontmatter. name, description, when_to_use, argument-hint, allowed-tools, model, effort, context: fork, agent, hooks, paths, disable-model-invocation, user-invocable.

Skill substitutions. $ARGUMENTS, $N, ${CLAUDE_SKILL_DIR}, ${CLAUDE_SESSION_ID}. Dynamic: !`bash` for shell preprocess.

Skill precedence. Enterprise > Personal (about /.claude/skills/) > Project (.claude/skills/). Plugin skills auto-namespaced as plugin:skill.

context: fork. Run skill in isolated subagent (built-ins: Explore, Plan, general-purpose). Prevents verbose output from polluting main context.

Sub-agents. .claude/agents/<name>.md. Frontmatter: tools, disallowedTools, model, permissionMode, mcpServers, hooks, maxTurns, skills, memory, effort, background, isolation, color.

Plugins. Manifest <plugin>/.claude-plugin/plugin.json. Components at plugin ROOT: skills/, agents/, hooks/hooks.json, .mcp.json, bin/. Use ${CLAUDE_PLUGIN_ROOT}.

.claude/rules/. Modular rule files with optional YAML paths: glob frontmatter (e.g. **/*.test.tsx). Loads only when matching files touched.

Plan vs direct. Plan mode: large multi-file architectural decisions, multiple valid approaches. Direct execution: single-file, well-scoped changes.

CI flags. -p / --print for headless. --bare skips local config (recommended CI). --output-format json + --json-schema then structured_output field.

Permission modes. default, acceptEdits (edits + filesystem commands such as mkdir/touch/mv/cp; other shell commands need explicit allow rules), plan, auto, dontAsk (strictest CI), bypassPermissions.

Diagnose team inconsistency. Check if instructions are in about /.claude/ (user-only) instead of project-level. Use /memory to verify loaded files.

Skill vs sub-agent. Skill = instructions injected into CURRENT context. Sub-agent = separate Claude with its OWN context. Bridge: context: fork on a skill spawns a subagent.

Allowed-tools per skill. allowed-tools: ['Bash(git add *)', 'Read'] in frontmatter pre-approves while skill active. To DENY use permissions.deny in settings.

Self-review. Independent reviewer instance catches more than the generator reviewing itself. Same session retains reasoning context.

CI re-run dedup. On re-runs after new commits, pass prior review findings into context and instruct Claude to report only new or still-unaddressed issues. Same pattern for test generation: pass existing test files in context to avoid duplicate scenarios.

Team config in about /.claude/. User-level config is NOT shared. Put team-shared content in .claude/ at project root.

Dir-level CLAUDE.md for spread files. Test files spread across dirs need .claude/rules/ with globs, not directory-bound CLAUDE.md.

Section three, question and answer review.

Scenario, Code Generation with Claude Code. Question, A developer's project has CLAUDE.md at the repo root, .claude/CLAUDE.md, about /.claude/CLAUDE.md, and a managed policy CLAUDE.md at /Library/Application Support/ClaudeCode/CLAUDE.md. When a session starts in the repo root, how are these files combined into context? Answer, All four files are concatenated together so they all enter context. Why, All discovered CLAUDE.md files are concatenated into context rather than overriding each other. Managed, user, project (./CLAUDE.md or .claude/CLAUDE.md), and CLAUDE.local.md all load together, ordered from filesystem root down to the working directory.

Scenario, Developer Productivity Tools. Question, A team wants every developer's machine to have one organization-wide CLAUDE.md that no individual user setting can suppress. On macOS, where must this file be deployed? Answer, /Library/Application Support/ClaudeCode/CLAUDE.md. Why, On macOS, the managed-policy CLAUDE.md location is /Library/Application Support/ClaudeCode/CLAUDE.md. Linux/WSL uses /etc/claude-code/CLAUDE.md, Windows uses C:\Program Files\ClaudeCode\CLAUDE.md. Managed CLAUDE.md cannot be excluded by individual settings.

Scenario, Code Generation with Claude Code. Question, A CLAUDE.md file uses @docs/git-instructions.md to import another file, which itself contains @docs/sub.md, and so on. What is the maximum depth of recursive @-imports Claude Code will follow? Answer, 5 hops. Why, Imported files can recursively import other files, with a maximum depth of five hops.

Scenario, Developer Productivity Tools. Question, In a large monorepo, a developer wants Claude Code to skip the CLAUDE.md files belonging to other teams in ancestor directories, while still loading their own. Which setting accomplishes this? Answer, claudeMdExcludes in settings.local.json with glob patterns. Why, claudeMdExcludes accepts an array of paths or glob patterns matched against absolute file paths and skips matching CLAUDE.md files. Managed-policy CLAUDE.md files cannot be excluded.

Scenario, Code Generation with Claude Code. Question, Auto memory was introduced in Claude Code v2.1.59+. Where does Claude Code store auto memory for a given project by default? Answer, about /.claude/projects//memory/MEMORY.md. Why, Each project gets its own memory directory at about /.claude/projects//memory/. The path is derived from the git repository, so worktrees and subdirectories share one auto memory directory.

Scenario, Code Generation with Claude Code. Question, How much of the auto memory MEMORY.md file is loaded at the start of every session? Answer, The first 200 lines or 25KB, whichever comes first. Why, The first 200 lines of MEMORY.md, or the first 25KB, whichever comes first, are loaded at the start of every conversation. CLAUDE.md files are loaded in full regardless of length.

Scenario, Developer Productivity Tools. Question, A developer wants per-project personal preferences (their sandbox URLs, test data) to load alongside team CLAUDE.md, but never be checked into git. Which file is the right place? Answer, ./CLAUDE.local.md (added to .gitignore). Why, CLAUDE.local.md at the project root is for private per-project preferences; it loads alongside CLAUDE.md and should be added to .gitignore.

Scenario, Code Generation with Claude Code. Question, After running /compact, a developer notices that an instruction from a nested CLAUDE.md in a subdirectory is gone, but the project-root CLAUDE.md instructions are still in effect. Why? Answer, Project-root CLAUDE.md is re-injected after compaction; nested CLAUDE.md files reload only when Claude next reads a file in that subdirectory. Why, Project-root CLAUDE.md survives compaction: after /compact, Claude re-reads it from disk and re-injects it. Nested CLAUDE.md files reload the next time Claude reads a file in that subdirectory.

Scenario, Developer Productivity Tools. Question, A team wants instructions that apply only when Claude is reading TypeScript files under src/api/. Which Claude Code primitive is the best fit? Answer, A path-scoped rule in .claude/rules/ with paths frontmatter glob. Why, Rules in .claude/rules/ support YAML frontmatter with a paths field containing glob patterns. Path-scoped rules are loaded only when Claude reads files matching the pattern.

Scenario, Code Generation with Claude Code. Question, Which command analyzes the codebase and generates a starting CLAUDE.md with build commands, test instructions, and conventions? Answer, /init. Why, Run /init to generate a starting CLAUDE.md automatically. Claude analyzes the codebase and creates a file with build commands, test instructions, and project conventions.

Scenario, Developer Productivity Tools. Question, Where on disk is a personal skill named 'commit' stored so it is available across all of a developer's projects? Answer, about /.claude/skills/commit/SKILL.md. Why, Personal skills live at about /.claude/skills//SKILL.md and apply across all projects.

Scenario, Code Generation with Claude Code. Question, A skill called 'deploy' exists at the enterprise, personal, and project scopes with the same name. Which one wins? Answer, Enterprise (managed). Why, When skills share the same name across levels, the precedence is Enterprise > Personal > Project. Plugin skills use a plugin-name:skill-name namespace and never conflict with other levels.

Scenario, Developer Productivity Tools. Question, A skill at .claude/skills/ship/SKILL.md and a custom command at .claude/commands/ship.md both exist in the same project. Which is invoked when the user types /ship? Answer, The skill, since skills take precedence over commands of the same name. Why, If a skill and a command share the same name, the skill takes precedence. Custom commands have effectively been merged into skills.

Scenario, Code Generation with Claude Code. Question, In a SKILL.md, what does the frontmatter field 'context: fork' do? Answer, Runs the skill in a forked subagent context, with the SKILL.md content as the subagent's prompt. Why, context: fork runs the skill in an isolated subagent context. The skill content becomes the prompt that drives the subagent, which doesn't have access to the conversation history.

Scenario, Multi-Agent Research System. Question, A skill is configured with 'context: fork' and 'agent: Explore'. What characterizes the Explore built-in subagent? Answer, Runs on Haiku with read-only tools (no Write/Edit) for fast codebase search. Why, The Explore subagent is fast and read-only, optimized for searching and analyzing codebases. It uses Haiku and is denied Write/Edit.

Scenario, Code Generation with Claude Code. Question, In a SKILL.md, what is the effect of setting 'disable-model-invocation: true'? Answer, Only the user can invoke the skill; Claude won't load it automatically. The skill's description is not added to context. Why, disable-model-invocation: true means only you can invoke the skill via /name; Claude won't load it automatically. Use this for workflows with side effects like /commit or /deploy.

Scenario, Developer Productivity Tools. Question, A team wants a skill that holds background context that Claude should know about when relevant, but should not appear in the user's / menu. Scenario, Code Generation with Claude Code. Question, In a SKILL.md, the user runs /migrate-component SearchBar React Vue. What does $1 expand to? Answer, React. Why, $N is shorthand for $ARGUMENTS[N], using 0-based indexing. So $0 = SearchBar, $1 = React, $2 = Vue. $ARGUMENTS always expands to the full argument string.

Answer, Claude Code executes git diff HEAD before sending content to Claude, replacing the line with the command's output. Scenario, Code Generation with Claude Code. Question, How does the permission rule `Bash(git diff *)` differ from `Bash(git diff*)`? Answer, `Bash(git diff *)` matches commands starting with 'git diff ' (with a space); `Bash(git diff*)` would also match 'git diff-index'. Why, The trailing ` *` (with space) enables prefix matching for commands starting with 'git diff '. Without the space, would also match 'git diff-index'.
